<p>현재 팀원들이 설계한 웹서비스 요건</p>
<ul>
<li>React 프론트엔드</li>
<li>웹소켓 실시간 채팅</li>
<li>Python(Flask/FastAPI) 백엔드</li>
<li>메시지 저장 (DB)</li>
<li>XSS 등 기본 보안 처리 -&gt; 클라이언트/서버 구조로 반드시 존재해야 함</li>
</ul>
<p>운영/보안 요건</p>
<ul>
<li>HTTPS</li>
<li>private DB</li>
<li>기본 모니터링</li>
<li>로그 남기기</li>
</ul>
<hr/>
<h2 id="aws-스택-조사">AWS 스택 조사</h2>
<h3 id="1-ec2">1. EC2</h3>
<ul>
<li>커스텀 서버 환경을 구성해야 할 때</li>
<li>WebSocket, 지속 연결 서비스에 적합</li>
<li>백엔드 서버 + 웹소켓 서버 + 프론트엔드 호스팅까지 전부 가능</li>
</ul>
<ul>
<li>웹소켓 장기 연결 유지가 필요할 때 사용</li>
<li>도커 기반 배포하고 싶을 때</li>
</ul>
<h3 id="2-alb-application-load-balancer">2. ALB (Application Load Balancer)</h3>
<ul>
<li>HTTPS/HTTP/WS/WSS 모두 지원</li>
<li>SSL 인증서(ACM)를 ALB에서 오프로딩 가능</li>
<li>s2n로 공격 테스트 시 ALB 로그가 유용함</li>
<li>Websocker은 API Gateway Standard HTTP API보다 ALB가 훨씬 안정적임</li>
</ul>
<ul>
<li>서비스가 HTTPS 필요할 때</li>
<li>로드밸런싱 필요할 때</li>
</ul>
<h3 id="3-vpc-public--private-subnet">3. VPC (Public + Private Subnet)</h3>
<ul>
<li>보안 영역 분리</li>
<li>DB를 Public에 둘 이유가 없음</li>
<li>EC2를 pricate로 넣고 ALB만 public에 두면 안전함</li>
</ul>
<ul>
<li>웹서비스에 DB가 포함될 때는 기본적으로 사용해야 함</li>
<li>Compliance 테스트하고 유리함</li>
</ul>
<h3 id="4-rds-mysql">4. RDS (MySQL)</h3>
<ul>
<li>관리형 DB -&gt; 백업/모니터링/스탭샷 자동</li>
<li>챗 메시지를 저장하려면 영속성 있는 DB 필요</li>
</ul>
<ul>
<li>팀원별로 DB에 접근해야 할 때</li>
<li>비용 최소화할려면 db.t3.micro / db.t4g.micro</li>
</ul>
<h3 id="5-cloudwatch-logs--metrics--alarms">5. CloudWatch (Logs + Metrics + Alarms)</h3>
<ul>
<li>웹소켓 연결 로그, 에러 로그, ALB 5xx 등 추적 가능</li>
<li>스캐너 공격할 때 어떤 패턴인지 모니터링 가능</li>
</ul>
<ul>
<li>로그를 중앙에서 관리해야 할 때</li>
<li>보안 테스트의 증거를 남길 때</li>
</ul>
<h3 id="6-s3">6. S3</h3>
<ul>
<li>ALB Access Logs 저장</li>
<li>백업된 채팅 로그 아카이빙</li>
<li>서버 로그 보관</li>
</ul>
<ul>
<li>장기 로그 보관이 필요할 때</li>
<li>비용 줄이기 위해 Glacier 자동 이동 활용해야 함</li>
</ul>
<h3 id="7-lambda">7. Lambda</h3>
<ul>
<li>DB 백업/정리 작업을 자동화</li>
<li>로그 압축, 정리 등 정기 작업 수행</li>
</ul>
<ul>
<li>서버리스로 정기 관리 작업을 처리하고 싶을 때</li>
<li>작은 운영 작업을 자동화할 때 비용 효율적</li>
</ul>
<h3 id="8-iam">8. IAM</h3>
<ul>
<li>인프라 운영 팀원별로 권한을 다르게 줄 수 있음</li>
<li>실수로 RDS 삭제/ALB 삭제같은 위험한 행동 방지 가능</li>
</ul>
<ul>
<li>실제 운영 환경처럼 운영하고 싶을 때</li>
</ul>
<hr/>
<h2 id="최종-설계-초안">최종 설계 초안</h2>
<h3 id="핵심-구성">[핵심 구성]</h3>
<ul>
<li>Route53</li>
<li>ALB (HTTPS + WebSocker 지원)</li>
<li>Private Subnet의 EC2</li>
<li>Private Subnet의 RDS (MySQL)</li>
<li>CloudWatch Logs</li>
<li>S3 (ALB Access Logs)</li>
<li>Secrets Manager (DB Credentials)</li>
</ul>
<p><img alt="" src="/assets/images/hosooinmymind/images/hosooinmymind/post/283c753f-9b85-47fe-8b0b-38201564e607/image.png"/></p>
<ul>
<li><p>WebSocket 통신의 안정성 때문에 ALB + EC2 선택</p>
<ul>
<li>WebSocket을 API Gateway로 처리하는 것보다 ALB WebSocker이 훨씬 안정적임</li>
<li>EC2에 WebSocket 서버 구동 자연스러움</li>
</ul>
</li>
<li><p>공격 테스트 목적 -&gt; 서버리스 이벤트 스트림 필요 없음</p>
<ul>
<li>Firehouse -&gt; OpenSearch -&gt; Athena는 운영/분석/SIEM 성능 강화형 아키텍처</li>
<li>목적이 ELK/ES 기반 분석이 아니라 취약점 공격 환경을 재현하는 것이므로 서버 로그는 CloudWatch로 충분하다고 생각됨</li>
<li>ALB Access Logs만 S3에 보존하면 s2n 테스트에 충분</li>
</ul>
</li>
<li><p>비용 고려</p>
<ul>
<li>Lambda + EventBridge + SNS + Firehouse + OpenSearch 등 사용하면 EC2/RDS보다 비용이 훨씬 올라감. OpenSearch는 최소 $75~150</li>
<li>EC2 (t3.micro) 1대 + RDS 1대 + ALB 1대 + ClouWatch &amp; S3 Access Logs</li>
</ul>
</li>
<li><p>XSS, SQLi, CSRF, File Upload 등 테스트를 하기 위해서는 Lambda로 하기는 어려움. EC2만이 자연스럽게 웹 취약점 발생, 웹소켓 유지 등 처리 가능</p>
</li>
</ul>
